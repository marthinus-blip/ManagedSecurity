# ManagedSecurity Database Schema Guidelines

## 1. Core Principles (Applies to All Databases)
*   **Immutable Timestamps:** Every entity must inherently have a `CreatedAtEpoch` (UTC `Int64` / `bigint`) column. This acts as the unforgeable Ground Truth for telemetry and data lifetime.
*   **Append-Only Strategy:** Tables must **never** be dropped. To remove visibility of data, implementations should utilize strategies like an `IsDeleted` or `IsDeprecated` flag.
*   **Zero Magic Values:** Migrations and ADO.NET query bindings must structurally map variables defensively, utilizing explicit constants (e.g., `JobIdQl`) following the `governance.md` definitions.
*   **Many-to-Many Tenant Hierarchy:** User identity is decentralized via a `TenantUserAccess` junction table, allowing users to adopt different context-specific roles depending on which `TenantId` they are actively proxying.

## 2. SQLite (`open_proj` / Edge Scouts) Constraints
SQLite is utilized exclusively as the embedded, isolated persistence store for edge hardware (`open_proj`), maintaining local state, task lease tracking, and agent configs.
*   **Atomic Definitions:** SQLite strictly limits `ALTER TABLE` capabilities (specifically regarding foreign keys and unique constraints). Therefore, absolute table structures (`CREATE TABLE IF NOT EXISTS`) containing all columns, constraints, and foreign keys must be holistically applied upon initialization.
*   **Separation of Application Layer Access:** SQLite is a serverless flat-file `.db`. Conceptually, the API process running the C# daemon holds total logical `RW` access. Application security is guaranteed by ASP.NET Core Middleware executing declarative Role-Based Access Control logic *before* instantiating internal SQLite queries.

## 3. PostgreSQL (`com_proj` / Commercial Tier) Constraints
PostgreSQL handles the scalable, multi-tenant Dashboard/API tier where security boundaries cross multiple distinct enterprise boundaries.
*   **Tenant Isolation Strategy:** We explicitly employ a **Shared Schema combined with Row-Level Security (RLS)**.
*   **Zero-Trust Guarantee:** Every table implementing `TenantId` must execute `ENABLE ROW LEVEL SECURITY`. 
*   **Connection Middleware Implementation:** 
    1. C# API extracts the JWT `ActiveTenantId` claim.
    2. Connection factory leases a Postgres connection and immediately executes `SET LOCAL app.current_tenant_id = '@id'`.
    3. The SQL engine naturally rejects any query attempting to cross the namespace boundary without relying on explicit `WHERE` clauses in domain code.
*   **Modifying Schemas:** Following standard ANSI SQL practices, mutations occurring to columns, non-RLS indices, or constraints *post-v1* are structurally applied via discrete `ALTER TABLE` migrations safely.
*   **Least Privilege Execution (Grant vs RLS Mitigation):** Since RLS inherently ignores whole-table operations (`TRUNCATE`) and metadata constraints (`REFERENCES`), the runtime ADO.NET connection must authenticate as a highly restricted role (e.g., `sentinel_api`).
    *   **Action Bounds (`GRANT`):** The C# role is explicitly granted *only* DML Tuple access (`SELECT`, `INSERT`, `UPDATE`, `DELETE`). It is completely stripped of DDL rights, `TRUNCATE`, or `REFERENCES` privileges to securely seal RLS bypass vectors.
    *   **Data Bounds (RLS):** Once the action query is authorized, RLS asserts the row visibility against the active `TenantId`.
    *   **Migration Separation:** All schema mutations (`CREATE TABLE`, Foreign Keys) must execute out-of-band using an isolated CI/CD Database Owner role (`sentinel_migrator`).
