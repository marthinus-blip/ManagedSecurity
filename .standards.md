# ManagedSecurity Database Naming Standards

## Principle: The Aesthetic of Verifiable Truth
Naming conventions in this repository forcefully reject legacy "Hungarian Notation" (such as prefixing tables with `tbl_` or procedures with `usp_`). Instead, we rely on semantic structural context and engine metadata to describe object behavior natively.

---

### 1. Tables
*   **Standard:** `PascalCase`, strictly pluralized nouns without any prefix.
*   **Examples:** `Cameras`, `Users`, `JobLeases`
*   **Why:** 
    *   **Contextual Redundancy:** SQL inherently proves structural context via its grammar (`SELECT * FROM {Table}`). Adding `tbl_` is visual noise.
    *   **Schema Segregation:** We utilize PostgreSQL Schemas (`auth.Users`, `sentinel.Cameras`) for compartmentalization natively, negating flat-namespace conflicts.
    *   **C# Mapping:** `PascalCase` maintains a flawless 1:1 conceptual mapping between our static C# dictionary definitions (`nameof(CameraRecordRl)`) and our ADO.NET query configurations (`const string CameraTableQl = "Cameras"`), eliminating all translation friction natively.

### 2. Stored Procedures / Functions
*   **Standard:** `snake_case`, strictly lowercased action verbs without prefixes.
*   **Examples:** `archive_camera_feed()`, `revoke_tenant_access()`
*   **Why:** 
    *   **Legacy Eradication:** The prefix `usp_` (User Stored Procedure) is a Windows legacy artifact previously used to avoid colliding with `sp_` system routines. PostgreSQL cleanly isolates engine primitives from user functions syntactically, rendering `usp_` entirely moot.
    *   **Execution Grammar:** Stored procedures execute distinct operational behaviors natively inside the engine; `snake_case` visually contrasts them from Entity Tables structurally.

### 3. Views
*   **Standard:** Prefix with `vw_` combined with `PascalCase` conceptual nouns.
*   **Examples:** `vw_ActiveAgents`, `vw_OfflineCameras`
*   **Why:** 
    *   A View *must* be distinguished from a standard Table structurally because it is fundamentally **immutable** via standard `INSERT` or `UPDATE` tuples physically. Prefixing with `vw_` acts as a developer warning that the bounded DML object is purely associative computation.

### 4. Structural Constraints & Indexes
Because PostgreSQL requires Indexes and Constraints to be unique strings within deeply congested global namespaces natively, they **MUST** securely implement explicit tracking-prefixes derived directly from their dependencies.

*   **Primary Keys:** `pk_TableName` (e.g., `pk_Tenants`)
    *   **Why:** Formally identifies physical key constraints distinctly across migrations natively.
*   **Foreign Keys:** `fk_SourceTable_TargetTable` (e.g., `fk_Cameras_Tenants`)
    *   **Why:** Explicitly defines referential bindings required for cascading wipe evaluations natively.
*   **Unique Constraints:** `uq_TableName_ColumnName` (e.g., `uq_Users_EmailAddress`)
    *   **Why:** Standardizing constraint naming empowers C# ADO.NET exception trapping (`NpgsqlException`) to easily substring parse exact boundary violations dynamically at runtime.
*   **Indexes:** `idx_TableName_ColumnName` (e.g., `idx_Cameras_LastSeenEpoch`)
    *   **Why:** Visually isolates and categorizes operational B-Tree structures when diagnosing raw Postgres `EXPLAIN ANALYZE` readouts structurally.

## Data Integrity & Performance Boundaries
The foundational architecture relies heavily on Append-Only isolation strategies natively. To ensure long-term stability and resilience against telemetry scale, the following limits are explicitly enforced:

### 1. The Soft-Delete Collision Rules (Partial Indexes)
*   **Standard:** Unique Constraints on natively soft-deleted entities must be formulated as Partial Indexes in PostgreSQL.
*   **Example:** `CREATE UNIQUE INDEX uq_Users_EmailAddress ON auth.Users (EmailAddress) WHERE (IsDeleted = false);`
*   **Why:** Because Tables are never dropped or records fundamentally `DELETED`, applying a standard `UNIQUE` constraint over a re-registered entity (e.g., a returning user email) triggers a catastrophic index collision against the deprecated row natively. Adding the `WHERE` clause structurally segregates the conflict cleanly.

### 2. Time-Series Sequence Exhaustion & Index Fragmentation
*   **Standard:** All structural primary keys and foreign keys must default to `Int64` (`BIGINT` / `BIGSERIAL`) natively. If decentralized edge generation is required (Scouts generating IDs locally before syncing), `UUIDv7` must be explicitly utilized structurally.
*   **Disallowed:** `INT` (Int32 caps at 2.1 billion records) and standard `UUIDv4`.
*   **Why:** Continuous edge camera telemetry flows exceed `Int32` maximums mathematically. Supplying purely random strings (`UUIDv4`) destroys PostgreSQL B-Tree insert mechanics natively, inducing massive Disk I/O page thrashing because new records are written arbitrarily across the disk span rather than temporally at the head.

### 3. The Audit Mutation Ledger
*   **Standard:** All mission-critical tables (`Tenants`, `Users`, `Cameras`) must structurally demand `UpdatedAtEpoch` (Int64) and an `UpdatedByUserId` constraint mapping securely.
*   **Why:** `CreatedAtEpoch` supplies the existential Ground Truth of creation, but yields zero forensic accountability over malicious configuration rewriting later within the lifespan. A managed security ecosystem mandates a contiguous mutation ledger natively.

### 4. RLS Architectural Hardening (The Connection Firewall)
To guarantee that developers cannot accidentally bypass Row-Level Security (RLS) by instantiating raw `NpgsqlConnection` objects outside of the `SentinelPostgresConnectionFactory`, three synergistic defense layers are unconditionally enforced:
1.  **Connection String Scrubbing:** The connection string parameter `Persist Security Info=false` is mandatory natively. This zeroes out the password in ADO.NET RAM the moment `OpenAsync()` executes, physically rendering extracted `DbConnection.ConnectionString` properties useless for launching rogue secondary connections natively.
2.  **Dependency Inversion Isolation:** Connecting strings must NEVER be mapped as arbitrary variables within `IServiceCollection` globally. They must be consumed deeply by the Factory singleton during `Program.cs` compilation natively.
3.  **Roslyn Compiler Enforcement:** A custom diagnostic analyzer (`MSG002`) will mandate a fatal compiler error if the abstract syntax tree detects `new NpgsqlConnection` or `new SqliteConnection` anywhere outside the approved `ISentinelDbConnectionFactory` boundaries physically.
